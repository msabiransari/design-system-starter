Overview

This is a framework-agnostic, token-based design system. It is not a component library like Material or Bootstrap. Instead, it is a theming contract: a set of semantic CSS custom properties that any application, component library, or UI framework can reference.

Apps reference --surface-default, not #ffffff. Swap one CSS file and the entire application re-skins.

The system ships with 8 built-in themes, motion tokens, elevation shadows, and optional primitive CSS classes (buttons, cards, inputs, tables). You can use the primitives directly, or reference tokens in your own components.

Quick Start

Load these 3–5 CSS files in your app's entry point:

<!-- 1. The token contract -->
<link rel="stylesheet" href="tokens.css" />

<!-- 2. A concrete theme (swappable at runtime) -->
<link rel="stylesheet" href="themes/professional.css" id="theme-link" />

<!-- 3. Base reset + globals -->
<!-- Use base.css for greenfield apps. Use base-scoped.css for legacy/hybrid apps. -->
<link rel="stylesheet" href="base.css" />

<!-- 4. Motion / animations (optional) -->
<link rel="stylesheet" href="motion.css" />

<!-- 5. Primitive classes (optional) -->
<link rel="stylesheet" href="primitives.css" />

Then build your UI using tokens or primitives:

<!-- Using primitives -->
<button class="ds-button ds-button-primary">Save</button>

<!-- Using tokens directly -->
<div style="background: var(--surface-default); border: 1px solid var(--border-default);">
  Custom component
</div>

Architecture

The system has three layers:

  1. Tokens — The semantic contract. Declares variables like --text-primary and --shadow-md but assigns no values.
  2. Themes — Concrete values. Each theme file defines what --text-primary actually is (e.g. #0a0e1a in Professional, #f9fafb in Dark).
  3. Primitives — Optional CSS classes that reference tokens. Drop-in buttons, cards, inputs, and tables.

This separation means:

  • Your app never touches hex codes.
  • You can add a new brand theme without changing a single line of markup.
  • Third-party libraries can be adapted to speak the same token language.

Library Adapters

If you use PrimeNG, Angular Material, or similar libraries, you do not need to fork them. Load an adapter CSS file after the library's theme. The adapter re-maps the library's internal CSS variables to your tokens. 

<!-- PrimeNG base theme -->
<link rel="stylesheet" href="primeng-theme.css" />

<!-- Adapter: bridges PrimeNG → your tokens -->
<link rel="stylesheet" href="adapters/primeng-adapter.css" />

Available adapters:

LibraryAdapterHow it works
PrimeNG (v17+)primeng-adapter.cssOverrides --p-primary-color, --p-surface-*, --p-content-*, and component classes.
Angular Materialmaterial-adapter.cssOverrides MDC token variables and component shim classes.

When you switch themes, PrimeNG buttons, tables, dialogs, and menus update automatically because they now reference your CSS custom properties.

Creating a New Theme

Copy any existing theme file and redefine the --_* variables inside :root:

/* themes/mybrand.css */
:root {
  --_surface-page:  #fafafa;
  --_surface-default: #ffffff;
  --_accent-primary: #d946ef;
  --_accent-primary-hover: #c026d3;
  /* ... */
}

You only need to redefine variables you want to change. Everything else inherits from the token contract.

For Platform Teams

If you build components consumed by other teams, treat tokens as your only source of truth:

  • New components reference tokens directly (e.g. background: var(--accent-primary)).
  • Old components get a shim CSS file that overrides their hardcoded values with tokens.
  • Add a stylelint rule banning hex colors in new files.
  • Publish tokens as an npm package (@acme/design-tokens) so apps import rather than copy.

The token system becomes the lingua franca of your organization. PrimeNG speaks it through an adapter. Your old libs speak it through a shim. Your new components speak it natively.

Legacy & Hybrid Apps

AngularJS + Angular 7 hybrid. Scattered CSS. Inline styles everywhere. Can this system actually help? Yes. But let’s be honest about the limits.

The Honest Assessment

ProblemSeverityFixable?

Inline styles (style="color: #333")HighPartially — must refactor
Inline styles with !importantCriticalNo — must refactor

Why It Works Despite the Mess

CSS custom properties are global and framework-agnostic. Load them in your index.html before any framework boots, and both AngularJS directives and Angular 7 components inherit the same variables. 

<!-- index.html -->
<head>
  <!-- 1. Tokens (both frameworks inherit these) -->
  <link rel="stylesheet" href="tokens.css" />
  <link rel="stylesheet" href="themes/professional.css" id="theme-link" />

  <!-- 2. Your scattered legacy CSS -->
  <link rel="stylesheet" href="legacy-app.css" />

  <!-- 3. AngularJS + Angular 7 bootstrap here -->

  <!-- 4. SHIM: overrides scattered legacy CSS -->
  <link rel="stylesheet" href="hybrid-shim.css" />
</head>

The Shim Layer

hybrid-shim.css is a tactical override file that targets known legacy classes and forces them to use tokens. Load it last so it wins the cascade. 

/* hybrid-shim.css */

/* Old AngularJS classes */
.legacy-btn,
.angularjs-button,
.btn-primary {
  background-color: var(--accent-primary) !important;
  color: var(--accent-primary-text) !important;
  border-radius: var(--radius-base) !important;
}

/* Old Angular 7 component classes */
.app-old-card,
.legacy-panel {
  background: var(--surface-raised) !important;
  border: 1px solid var(--border-default) !important;
  box-shadow: var(--shadow-sm) !important;
}
Yes, we use !important here. The shim is a demolition tool, not a sculpture. It is temporary. Its only job is to make old UI look themed while you refactor.

Inline Styles: The Hard Boundary

If you have style="color: #333" or [ngStyle]="{'color': '#333'}" in templates, CSS cannot override that from the outside. Do not try to !important every possible inline property — it is madness.

Strategy for inline styles:

  1. Don't theme them from the outside. Accept they will look wrong until refactored.
  2. Use a Style Guard in CI. Fail the build if new/modified files contain inline colors: 
    // style-guard.js
const forbidden = /style=["'][^"']*(#[0-9a-f]|rgb\(|hsl\()/i;
// Reject commit if match found
  3. Migrate the top 20 offenders. Find the components users see most (nav, buttons, cards, tables). Refactor only those to token-based CSS classes. Leave obscure admin screens for later.
  4. Runtime helper (dev only). If you must prove the concept without touching code: 
    // DO NOT SHIP TO PRODUCTION
function themeInlineStyles() {
  document.querySelectorAll('[style*="rgb"], [style*="#"]').forEach(el => {
    const map = {
      'rgb(51, 51, 51)': 'var(--text-secondary)',
      '#ffffff': 'var(--surface-default)',
      // ... add your audit results
    };
    if (map[el.style.color]) el.style.color = map[el.style.color];
  });
}

Using Tokens in Both Frameworks

AngularJS (1.x):

angular.module('legacyApp').component('legacyCard', {
  template: `
    <div class="legacy-card"
         ng-style="{'background-color': 'var(--surface-default)'}">
      <h3 style="color: var(--text-primary)">{{ $ctrl.title }}</h3>
    </div>
  `
});

Angular 7:

@Component({
  selector: 'app-modern-card',
  template: `<div class="modern-card">{{ title }}</div>`,
  styles: [`
    .modern-card {
      background: var(--surface-raised);
      border: 1px solid var(--border-default);
      border-radius: var(--radius-md);
      padding: var(--gap-md);
      color: var(--text-primary);
    }
  `]
})
export class ModernCardComponent {}

Both frameworks read the same values because the CSS is global. Your hybrid is actually an advantage here — one token file themes both simultaneously.

Week 1Inject tokens + theme into index.html. Do nothing else.Verify both frameworks see the variables.
Week 2Create hybrid-shim.css with !important overrides for your 10 most common legacy classes. Load it last.App looks 70% themed immediately.
Week 3–4Add CI lint rule banning inline colors. Refactor top 20 visible components (nav, buttons, cards).Stop the bleeding. Users see the new theme.
Month 2–3As you touch files for feature work, replace hardcoded values with tokens.Strangler fig migration. No big bang.
Month 6+Remove shim rules as components are rebuilt. Eventually delete hybrid-shim.css.Clean system.

The Bottom Line

You don't need a clean codebase to use a design system. You need a clean boundary. The boundary is: from today forward, all new CSS uses tokens. Old CSS gets shimmed. Inline styles get refactored when touched.

Recommended Loading Order

1. tokens.css              ← semantic variables
2. themes/{name}.css       ← concrete values (swappable)
3. base.css                ← reset + global typography
   base-scoped.css         ← scoped reset for legacy/hybrid apps
4. motion.css              ← animations
5. primitives.css          ← optional utility classes
6. [library theme].css     ← PrimeNG / Material base
7. adapters/{lib}.css      ← bridges library → tokens
8. [app-specific CSS]      ← app pages + components
9. legacy-shim.css         ← old component overrides (temporary)
